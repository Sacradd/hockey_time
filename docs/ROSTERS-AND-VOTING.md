# Roster, голосование и состав (согласованная модель)

> Замена логики «одна группа = один день» на модель как в Telegram: **постоянный пул** → **игра на дату** → **голосование** → **состав**.

## Аналогия с Telegram

| Telegram | Приложение |
|----------|------------|
| Чат со «своими» | **Roster** (постоянная группа участников) |
| «Голосование в среду на Кристалл» | **Игра** (`day_group`) на конкретную дату |
| Кнопки / ответы в опросе | Варианты ответа (**задаёт админ** при старте) |
| Список «едут» | **Состав**: 20 полевых + 2 вратаря (отдельно) |

---

## 1. Несколько roster (сейчас одна, в планах много)

**Сейчас:** одна группа, например **«Среда · ЛД Кристалл»**.

**Позже:** отдельные roster без переделки архитектуры, например:

- Среда · ЛД «Кристалл»
- Пятница · ЛД «Ногинск» (ребята уже есть)

У каждого roster свой пул `roster_members`. Игры (`day_groups`) создаются **внутри** roster на дату.

```
rosters
  └── roster_members (user_id)
  └── day_groups (игра на дату)
        └── votes
        └── состав (расчётный)
```

---

## 2. Две «роли» у пользователя (не путать с admin)

| Поле | Значения | Смысл |
|------|----------|--------|
| `users.role` | `admin` \| `player` | Доступ в приложение (как сейчас) |
| `users.position` | `player` \| `goalie` | Тип в составе при голосовании |

### Полевые (`player`)

- Голосуют на игру.
- В общий список: **не более 20** по времени ответа «еду» (первые 20 — в игру, остальные — резерв).
- Лимит **20 всегда** для полевых.

### Вратари (`goalie`)

- Тоже голосуют (те же варианты ответа).
- **Не входят** в лимит 20 полевых.
- Отдельный список: **2 позиции** (первые два вратаря по времени голоса «еду»).

На экране игры два блока:

1. **В игру** (до 20 полевых) + резерв полевых  
2. **Вратари** (до 2)

---

## 3. Голосование: подписи ответов задаёт админ

При **запуске голосования** админ указывает тексты кнопок (как в Telegram-опросе), например:

- «Еду» / «Не еду»
- «Буду» / «Пропускаю»
- «Да» / «Нет» / «Под вопросом» (третий вариант — опционально)

В БД на игру (`day_groups`), например:

| Поле | Пример |
|------|--------|
| `vote_label_1` | Еду |
| `vote_label_2` | Не еду |
| `vote_label_3` | NULL или «Под вопросом» |
| `vote_go_option` | `1` — какой вариант считается «еду в состав» (обычно первый) |

Игрок видит **именно эти** подписи. В `votes` сохраняем выбор: `1`, `2` или `3`.

---

## 4. Поток админа

1. Roster **«Среда · ЛД Кристалл»** — добавить участников (телефон, пароль, `position`: player / goalie).
2. **Создать игру** на дату (следующая среда).
3. **Запустить голосование** → ввести подписи ответов + время окончания → push пулу (этап push позже).
4. Смотреть состав: 20 + резерв + 2 вратаря; при необходимости убрать из состава (этап 5).

## 5. Поток игрока / вратаря

1. Вход → видит активное голосование своего roster.
2. Жмёт один из вариантов (подписи от админа).
3. После закрытия голосования — видит, попал в основу / резерв / вратарскую пару или нет.

## 6. Схема БД (целевая, миграция с текущей)

**Новое / изменения:**

```sql
users
  + position ENUM('player','goalie') NOT NULL DEFAULT 'player'

rosters
  id, title, venue, weekday TINYINT NULL, created_at
  -- title: "Среда · ЛД Кристалл"

roster_members
  roster_id, user_id, created_at
  UNIQUE (roster_id, user_id)

day_groups  -- игра на дату
  + roster_id FK
  + vote_label_1, vote_label_2, vote_label_3 VARCHAR(64) NULL
  + vote_go_option TINYINT DEFAULT 1   -- какой ответ = "еду"
  vote_active, vote_ends_at, ...

votes
  user_id, group_id
  + choice TINYINT NOT NULL   -- 1, 2 или 3
  voted_at
  UNIQUE (user_id, group_id)

-- group_members / actual — устаревает для состава;
-- оставить для "аккаунт активирован" или убрать позже
```

**Расчёт состава (без отдельной таблицы на первом этапе):**

- Полевые с `choice = vote_go_option`, `position = player`, не excluded → ORDER BY `voted_at` → LIMIT 20 основа, остальные резерв.
- Вратари с тем же «еду», `position = goalie` → LIMIT 2.
