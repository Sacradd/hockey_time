## Рассуждения (кратко)

### Было: игрок сразу в одну группу

Создание через `create-player.php` всегда требовало `roster_id` — удобно для одной группы «Среда · Кристалл», но не совпадает с жизнью:

- **Super** не знает, когда будет лёд и как назовётся группа.
- **Админ** сам знает площадку и день недели — группу создаёт он.

### Стало (решение 2026-05)

1. **`users`** — общий пул аккаунтов (все игроки приложения).
2. **`rosters` + `roster_members`** — кто в какой группе; `is_admin` только внутри группы.
3. **Super** — только пул: телефон, пароль, опционально «будущий админ» (`users.role = admin` **без** привязки к группе).
4. **Админ** (`role = admin` после super) — создаёт **группы** (+ на главной), внутри группы добавляет игроков (новый в пул + связь или поиск по пулу).
5. **Свайп влево** в списке группы — отвязка от группы (аккаунт в пуле остаётся).
6. **Удаление из приложения** — только super (блок «Пользователи»).

### Что сознательно отложено

- Push при голосовании.
- Полное удаление игрока админом группы.
- Несколько групп у одного админа в UI (архитектура уже позволяет).
